# contract-platform

Генератор договоров для Медиа Лэнд: FastAPI + PostgreSQL + MinIO + React,
всё в Docker Compose на VPS. Русскоязычные юридические документы (.docx)
по шаблонам с метками, данные подставляются из карточек контрагентов.

Общение — по-русски.

## Деплой — КРИТИЧНО

**Автоматический.** В bare-репо на сервере (`~/git/contract-platform.git`) есть рабочий хук
`post-receive`, который сам раскладывает файлы в `~/contract-platform` после пуша.
**Ручной `git --git-dir=... --work-tree=... checkout` НЕ нужен** (раньше ошибочно
считался обязательным — это не так, проверено).

Два remote, **пушить всегда в оба**:
```bash
git push origin main && git push production main
```

Полный цикл правки:
```bash
# 1. если менялся фронт — пересобрать (иначе прод увидит старую сборку):
cd frontend && ./deploy.sh && cd ..

# 2. закоммитить И исходники, И сборку:
git add <файлы> backend/static/app
git commit -m "..."
git push origin main && git push production main

# 3. ТОЛЬКО если добавлена миграция:
ssh root@64.188.98.101 "cd ~/contract-platform && docker compose exec api alembic upgrade head"
# затем убедиться, что воркер перезапустился на новой схеме (см. ниже про окно):
ssh root@64.188.98.101 "cd ~/contract-platform && touch backend/app/main.py"
```

Что хук НЕ делает:
- **не накатывает миграции** — всегда отдельным шагом, вручную;
- не перезапускает контейнеры — не нужно: `backend/app`, `backend/static`, `backend/alembic`
  смонтированы как volume, `uvicorn --reload` подхватывает правки сам.

**Подводный камень: окно «код ↔ миграция» (инцидент 16.07.2026).** Пуш деплоит
код мгновенно (volume + `uvicorn --reload`), а миграцию накатывают отдельным
шагом ПОСЛЕ. Поэтому между пушем и `alembic upgrade` воркер уже работает на
новом коде со СТАРОЙ схемой БД: если новый код читает/пишет добавленную колонку,
он падает на старте (`Application startup failed. Exiting.`) до накатки миграции.
Reloader при этом жив (контейнер `Up`), но воркер не перезапустится сам, пока не
изменится watched-файл. Порядок для миграций, где код уже использует новое поле:
1. `git push …` (код уедет и воркер, возможно, временно упадёт — это ожидаемо);
2. `alembic upgrade head`;
3. `touch backend/app/main.py` — триггерит `--reload`, воркер поднимается на
   новой схеме. Проверить `docker compose logs --tail=6 api` → «Application
   startup complete». Для аддитивной nullable-колонки это лишь ~минута простоя;
   для несовместимых изменений схемы планировать деплой аккуратнее.

**Почему сборка React лежит в git.** На сервере нет Node — собрать там нечем.
`backend/static/app/` попадает в контейнер через уже существующий volume на
`backend/static`, поэтому ни `docker-compose.yml`, ни пересборка образа не нужны.
Это осознанное решение, а не недосмотр.

**Правило после инцидента 15.07.2026**: недоделанный React выложили поверх
`backend/static/index.html` и сломали вход на проде. Поэтому старый интерфейс живёт
на `/`, новый — на `/app`. Не заменять `index.html` без явной просьбы: это откат
в один клик, если что-то пойдёт не так.

## Роли

`app/roles.py` — единственный источник правды. `frontend/src/auth/permissions.js` —
его зеркало: при правке одного правь и второе.

| действие | admin | director | top_manager | manager |
|---|---|---|---|---|
| генерация, создание контрагента | + | + | + | + |
| редактирование карточки контрагента | + | + | + | – |
| удаление контрагента | + | – | – | – |
| экспорт в Excel | + | + | + | – |
| импорт из Excel | + | – | – | – |
| папки/шаблоны: просмотр | + | + | + | + |
| папки/шаблоны: управление | + | – | – | – |
| вкладка «Пользователи» | + | – | – | – |
| просмотр audit_log | + | + | – | – |
| вкладка «История генерации» | + | + | – | – |
| смена своего пароля | + | + | + | + |

`top_manager` — менеджер с расширенными правами: как `manager`, плюс
редактирование карточек контрагентов и экспорт в Excel. Без просмотра
audit_log (в отличие от `director`) — сознательно, см. `app/roles.py`.

Права на фронте — только UX (не показывать кнопку, которая вернёт 403).
Настоящая защита всегда на сервере: `Depends(require_role(...))`.

Админ **не может** сменить роль себе или деактивировать себя (400) — иначе
единственный админ запер бы себя снаружи безвозвратно.

## Ключевые принципы (нарушать только осознанно)

**Форма генерации строится из .docx, а не из кода.** `template_analysis.analyze_template()`
сканирует метки (`{{ x }}`, `{% for %}`, `{% if %}`) и выводит тип поля
(text/date/flag/choice/list). Добавили метку в шаблон → поле появилось в форме само.
В `DocFormPage.jsx` нет ни одного захардкоженного поля — так и должно остаться.
Поэтому новые типы контрагентов (ИП, ООО с НДС) не требуют правок фронта.

**maps_to — это предзаполнение, а не подстановка.** `GET /templates/{id}/fields?contragent_id=...`
кладёт значения из карточки в `default`. `POST /generate` их не проверяет и не
перезаписывает: единственный источник истины для документа — то, что реально пришло
в теле запроса. Оператор может поправить любое подставленное значение.

**Справочники — только с сервера.** Страны, типы контрагентов, типы договоров, роли,
`reg_number_meta` (ИНН/ОГРНИП/ОГРН + длины) приходят из `GET /tags`. Фронт их не
хардкодит: бэкенд теми же значениями валидирует, любое расхождение = 400 при сохранении.

**reg_number — одна колонка на три смысла.** ИНН (СГ, 12 цифр) / ОГРНИП (ИП, 15) /
ОГРН (ООО, 13). Какой именно — определяет поле `type`, отдельных колонок нет.
`unique=True` — это точный идентификатор контрагента. При смене статуса заводится
НОВАЯ карточка, значение живёт всё время жизни записи.

**title и contract_number вычисляет сервер**, руками не вводятся и не редактируются.

**Готовые документы нигде не хранятся.** `POST /templates/{id}/generate` рендерит
docx/pdf "на лету" и не кладёт результат ни в MinIO, ни куда-либо ещё (было
осознанное решение ещё на этапе 6 — см. `models.py`: раньше там был явный
комментарий "generated_documents сознательно НЕТ"). На этапе 7 это решение
пересмотрено под конкретную задачу — вкладку "История генерации": вместо файла
таблица `generated_documents` хранит payload формы + template_id, и результат
можно воссоздать по требованию тем же `render_document()`. Если шаблон с
момента генерации перезалили — воссозданный документ будет соответствовать
НОВОЙ версии шаблона, это ожидаемый компромисс (см. докстринг `GeneratedDocument`).

## Как всё устроено (то, что неочевидно из кода)

- **JWT**: access 30 мин, refresh 14 дней с ротацией при каждом использовании.
  В `client.js` есть замок на refresh — несколько параллельных 401 не должны дёргать
  `/auth/refresh` каждый своим токеном, иначе второй придёт с уже отозванным и
  выкинет на логин при живой сессии.
- **Тема**: light по умолчанию, ручное переключение, хранится в localStorage `ml_theme`.
- **Модалки**: стек, Escape закрывает только верхнюю (`isTop` из `ModalRoot`).
- **Никнеймы**: у контрагента их может быть несколько. Если `maps_to='contragent.nickname'`,
  сервер шлёт `nickname_options` (список) вместо одного default'а — фронт рисует select.
- **Парный Акт**: у Приложения, открытого по контрагенту, ищется Акт той же связки тегов
  и предлагается сформировать по тем же данным.
- **Подбор документов** контрагенту — строгое равенство трёх тегов
  (country/contragent_type/contract_family). Несовпадение регистра = документ просто
  не найдётся, без ошибки. Помнить при отладке.
- **Списки в форме**: пустые строки отбрасываются при сборке payload, но галочки
  (`has_profanity`, `is_group`) при проверке «строка пустая?» не учитываются — иначе
  строка с одной галочкой попала бы в документ пустой.

```bash
# фронт

npm install
npm run build          # сборка в dist/
npm run lint           # oxlint (4 warning'а про Fast Refresh — норма)
./deploy.sh            # build + копирование в backend/static/app/

# сервер
ssh root@64.188.98.101
cd ~/contract-platform
docker compose ps
docker compose logs -f api --tail=50
docker compose exec api alembic current
docker compose exec api alembic upgrade head
docker compose exec db bash -c 'psql -U contracts_app -d contracts -c "\d contragents"'
```

`docker compose`) — либо `bash -c '...'` внутри контейнера, либо значения руками.

## Чего не делать

- Не выкладывать React поверх `backend/static/index.html`.
- Не забывать `./deploy.sh` перед коммитом, если менялся фронт — иначе на проде
  останется старая сборка при свежих исходниках.
- Не пушить только в один remote.
- Не вызывать `init_db()`/`create_all()` на сервере — только alembic (уже был инцидент:
  create_all создал таблицы до миграции, и `alembic upgrade` упал на «relation already exists»).
- Не хардкодить на фронте то, что отдаёт `GET /tags`.
- Не добавлять поля формы генерации в код фронта — они приходят из .docx.
