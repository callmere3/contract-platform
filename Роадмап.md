# Роадмап contract-platform

Составлен 16.07.2026 по итогам аудита кодовой базы.

---

## Где проект сейчас

Основной сценарий закрыт полностью: контрагент → подбор шаблона → форма из .docx →
генерация → история с возможностью пересоздать документ.

В коде: 4 роли, 7 экранов, 8 таблиц, 6 миграций, 30+ эндпоинтов.

Сделано за последний месяц: роль `top_manager`, история генерации (2 этапа),
автовычисления в форме (срок действия, штраф, дата поставки), подсветка неполных
карточек, статусы «в сети», экспорт с учётом фильтров.

**Вывод для планирования:** фаза «сделать, чтобы работало» закончена. Дальше
конкурируют три разных типа работы — закрыть риски, сделать правки безопасными,
добавить возможности. Ниже они разведены по волнам: смешивать их — верный способ
не сделать ни одного.

## Риски (то, что пока просто не выстрелило)

### 1. Бэкапов нет — единственная невосстановимая вещь

Код лежит в git на двух remote. Шаблоны .docx есть на диске разработчика. А
**карточки контрагентов существуют ровно в одном экземпляре** — в Postgres на
одном VPS. В репозитории нет ни `pg_dump`, ни cron, ни единого упоминания бэкапа.

Умрёт диск, обнулит провайдер, выполнится не тот SQL — база уйдёт целиком, и
восстанавливать будет неоткуда. Всё остальное в этом документе — неприятности.
Это — конец проекта.

- **Сначала проверить:** есть ли у хостера снапшоты VPS (возможно, риск уже
  частично закрыт).
- **Если нет:** `pg_dump` по cron + автовыгрузка копии за пределы VPS.
- **Стоимость:** полдня.

### 2. Всё летит по открытому HTTP

Прод: `http://64.188.98.101:8000`. Пароль при логине и JWT-токены идут по сети
**открытым текстом**. Любой между пользователем и сервером — публичный wi-fi,
провайдер, сосед по гипервизору — читает пароль администратора как есть.
Rate limit на `/auth/login` защищает от перебора, но не от прослушки.

Записано в CLAUDE.md как блокер из-за DNS — и технически так и есть. Но задача не
техническая: нужно, чтобы кто-то с доступом к DNS `medialand.su` завёл A-запись на
`docs.medialand.su`. Как только домен появится — Caddy с автоматическим
Let's Encrypt поднимается за час.

**Единственный пункт роадмапа, упирающийся не в код.** Выяснять, к кому идти,
стоит начинать уже сейчас.

### 3. Веб-консоль MinIO смотрит в интернет

Порт `9001` проброшен наружу в `docker-compose.yml`. За ним — все шаблоны
договоров компании. Если там дефолтный или слабый пароль, их скачает любой
желающий.

Порт Postgres, к слову, уже закрыт — сделано правильно, MinIO просто забыли за
компанию.

- **Стоимость:** 10 минут (убрать проброс порта из compose).

### 4. Тестов нет ни одного, CI нет

Пока проект был маленьким — нормально. Сейчас в нём есть логика, которую глазами
не проверишь:

- расчёт штрафа (аванс ÷ количество треков);
- срок действия (+5 лет до конца квартала);
- дата предоставления исходников (+1 год до конца месяца);
- сортировка треков в сноске по значению НЛ первого трека;
- матрица прав из четырёх ролей;
- синхронизация исполнителей из таблицы треков в сноску.

Всё это — **чистые функции без БД**, идеальные кандидаты для тестов. Сегодня
единственный способ узнать, что правка что-то сломала, — заметить это на проде.
За последний месяц так поймали как минимум два случая (fio предзаполнялся не тогда,
когда надо; API упал из-за окна «код ↔ миграция»).

Это не про «правильность» — это про скорость. С тестами правки перестают быть
рулеткой, и следующие фичи едут заметно быстрее.

- **Стоимость:** 1–2 дня (pytest поверх `context_builder`/`roles` + GitHub Actions).

Делать первым. Это не улучшение продукта — это страховка от того, что продукта не
станет.

## Волна 1 — перестать бояться правок (~2 дня)

- Тесты на вычисления в `context_builder`.
- Тесты на матрицу ролей.
- Тест на расхождение `roles.py` ↔ `frontend/src/auth/permissions.js` — зеркало,
  которое легко забыть обновить, тест поймает автоматически.
- GitHub Actions: гоняет тесты на пуш + проверяет, что фронт собирается.
- **Отдельно:** проверка в CI, что `backend/static/app` не отстал от исходников.
  Это самая коварная грабля проекта — собранный фронт лежит в git, и забытый
  `./deploy.sh` означает свежий код в репозитории и старый интерфейс на проде,
  без единой ошибки где-либо.

## Волна 2 — то, что просят пользователи (~1–2 недели)

### Шаблоны ИП и ООО с НДС

Единственная фича, где архитектура уже готова и ждёт, а работы в коде нет вообще:
форма строится из .docx, справочники приходят с сервера, `reg_number` уже умеет
ОГРНИП/ОГРН. Нужны только сами файлы шаблонов с метками.

Пока их нет, целые категории контрагентов обслуживать нечем. **Самая дешёвая
бизнес-ценность в списке — узкое место не в разработке, а в том, чтобы юристы
дали .docx.**

### Пакет документов одним архивом

Сейчас договор, приложение и акт качаются по одному, каждый через свою форму.
Логика парного акта уже есть (`ContragentDocsModal`) — расширить до «сформировать
весь комплект в zip». Экономит менеджеру десятки кликов на каждом контрагенте.

### Предпросмотр перед скачиванием

Сейчас цикл «сформировал → скачал → открыл → увидел опечатку → вернулся, поправил»
проходится целиком ради каждой мелочи. Конвертер в PDF уже стоит отдельным
контейнером — показать результат прямо в браузере технически почти бесплатно.

## Волна 3 — новые возможности (обсуждаемо)

### Дашборд сроков

`term_end` и дата предоставления исходников уже вычисляются, но просто попадают в
документ и исчезают. Никто не узнает, что через месяц истекает договор, пока не
вспомнит сам.

Экран «что истекает в ближайшие 60 дней» превращает генератор документов в
инструмент, куда заходят не только чтобы что-то сгенерировать. **Самая интересная
фича по соотношению польза/усилия** — данные уже есть, нужен только запрос и экран.

### Мягкое удаление контрагентов

Сейчас `DELETE` физический и безвозвратный. История генерации удаление переживёт
(`SET NULL` + снимки полей — сделано правильно), но сама карточка со всеми
никнеймами исчезает навсегда, и восстановить её можно только из бэкапа, которого
нет. Флаг `deleted_at` + «Корзина» для админа стоит немного и убирает целый класс
«ой».

### Вёрстка DocForm по §6.1 дизайн-токенов

Из зафиксированного ранее плана. В `design-tokens-ml-docs.md` §6.1 заданы разные
сетки на секцию (4 колонки «Документ», 3 «Контрагент», 2 «Релиз»), а в
`DocFormPage.jsx` везде `grid-cols-3`. Требуется скриншот, чтобы понять, что именно
«поплыло».

## Рекомендация

1. **Начать с Волны 0** — буквально один день, снимает риск потерять всё.
2. **Параллельно запустить два процесса, не зависящих от кода:** узнать про DNS
   для HTTPS и запросить у юристов .docx для ИП/ООО. Оба упираются в других людей,
   поэтому чем раньше начаты, тем лучше.
3. **Пока они идут — Волна 1.** Дальше фичи поедут быстрее и спокойнее.
4. **Из содержательного после этого — дашборд сроков:** он ближе всего к тому,
   чтобы изменить характер продукта, а не просто добавить ещё одну кнопку.

- Мусорная строка `# test-hook-check` в корневом `.gitignore`.
- Документы проекта (`Контекст_проекта.md`, `design-tokens-ml-docs.md`,
  `Брейншторм_база_контрагентов.md`) лежат на директорию выше репозитория и в git не
  попадают, хотя на них ссылаются и CLAUDE.md, и комментарии в коде. Стоит решить:
  занести в репозиторий или оставить снаружи сознательно.
- Судьба старого `backend/static/index.html` — когда новый интерфейс окончательно
  подтвердится, решить, оставлять ли откат в один клик.
